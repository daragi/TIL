## **4️⃣ 질문–KPI 연결 정리**

BI 질문 1: 문의 처리 비용이 가장 많이 드는 기능은 무엇인가? 	

- 문의 처리 시간  
- 문의 횟수 (문의 재처리 비용)

BI 질문 2: 기능별로, 문의증가율이 가장 높은 기능은 무엇인가?

- 티켓 수  
- 심각도 High인 문의 건수   
* 기능별 사용자 수  
* 돈 \+ 리소스   
* 특정 이벤트 발생 이후 트래픽 \+ 문의 증가  
* 고객 불만 처리 핵심 성과 지표 (KPI)  
  * 반복 불만 건수 (1,000명당): 특정 문제로 인해 반복해서 들어오는 불만 건수로, 제품/서비스의 근본적 문제점을 나타냅니다.

## **5️⃣ 대시보드 구조 초안**

(화면을 위에서 아래로 본다고 가정)

필요한 대시보드 요소:

- ### 전체 문의 건수

- 월별 문의 건수 추이   
- 처리된 문의 수   
- 문의 재처리 횟수  
- 기능별 문의 처리 시간  
- 기능별 문제 심각도별 건수

### **\[상단\] 전체 판단 요약 영역**

- 문의 처리율 (=처리된 문의 건수/ 전체 문의 건수)   
  - 문의 처리가 되지 않아서 발생하는 비용 \=\> 비교 판단 영역을 통해 보여줌  
- 전체 문의 건수

### **\[중단\] 비교 판단 영역**

- 월별 문의 건수 추이  
- 기능별 문의 처리 시간

### **\[하단\] 상세 탐색 영역**

- 기능별 문제 심각도별 건수  
- 기능별 문의 재처리 횟수




**\[상단\] 전체 판단 요약**

* 처리율

* 전체 문의 건수

**\[중단\] 비교 판단**

* 월별 문의 건수 추이

* **기능별 문의량(또는 영향 점수 \= 문의량×처리시간)** ← *우선순위 닫는 키*

**\[하단\] 상세 탐색**

* 기능별 심각도 분포  
* 기능별 처리 시간  
* **재처리율(또는 재처리 건수 \+ 분모 표기)**

상단에서는 전체 문의 처리율과 문의량을 통해 운영이 정상인지, 병목이 발생했는지 빠르게 인식하고, 미처리로 인한 영향 규모를 통해 지금 조치가 필요한 수준인지 판단한다.  
⇒ 지금 운영이 괜찮은지 바로 상태 파악 

중단에서는 시간 흐름(월별 추이)과 기능 단위(기능별 문의량)를 비교해 문의 증가가 단순한 물량 문제인지, 특정 기능에서 처리 병목이 생긴 구조적 문제인지 원인을 좁혀간다.  
⇒ 범인 찾기: 무엇이 문제를 만들고 있는가

하단에서는 문제 후보 기능을 심각도, 처리시간과 재처리 지표로 상세 확인하여 “정말 우선적으로 개선할 기능”을 확정하고, 재작업이 큰 영역을 중심으로 개선 과제의 우선순위를 결정하게 된다.  
⇒ 진짜 어디부터 손대야 할지 액션 결정

**최종판단문장 \= “이번 주 개선 타겟은 A기능 (심각도 높고 재처리 많고 처리 시간 오래 걸림)”**

## **6️⃣ 이 기획서가 필요한 이유**

**고객지원팀 팀장이 문의에 따라 어떤 걸 우선적으로 개선해야 할지 보는 대시보드를 만들기 위함**

* 이 기획서가 없다면 대시보드의 목적이 기능 개선용인지, 상담원 관리용인지 명확히 구분되지 않아 설계 방향이 흔들릴 수 있음  
* ‘리소스’의 정의(비용, 처리시간, 인력 등)가 합의되지 않으면 같은 지표를 두고도 해석이 달라질 수 있음  
  →  지표는 많지만 실제 우선순위 판단에 활용되지 못하는 보고용 화면이 될 위험이 있음

  ⇒ 대시보드를 구현하기 전에 판단 기준과 설계 방향을 고정하기 위한 합의 문서

이 기획서가 없었다면, 이 화면은 ‘모니터링 대시보드’처럼 흘러가며, KPI→원인→액션으로 닫히는 판단 논리(지표 정의/우선순위 기준/액션 주체)가 약해져 상·중·하단의 연결이 흔들렸을 것
